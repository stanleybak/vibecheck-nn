# vggnet16_2022 — ImageNet VGG16

**Net:** `vgg16-7.onnx` (528 MB, input 3×224×224): 13 Conv + 5 MaxPool + 15 ReLU
+ 3 FC, 1000-class. **18 instances @ 1200 s.** 2026 instances are md5-identical
to 2025, so the α,β-CROWN (ABC) 2025 results are the direct reference.

## Status: VC 16/18 == ABC 16/18 (strict 1200 s) — spec15 newly closed

| group | instances | VC | ABC | notes |
|---|---|---|---|---|
| sparse-perturbation | spec0 (sat), spec1–14 (unsat) | **15/15 ✓** | 15/15 | spec0 = real CE (nominal misclassification, ORT-validated); spec1–14 unsat in 40–148 s (≈2–3× ABC's 15–41 s but correct) |
| full-image L∞ | spec15/16/17 | **1/3 ✓** | 1/3 strict | all 150528 px perturbed, ε 2e-7 / 2e-6 / 2e-5. **spec15 now `unsat` in 1104 s (in budget)** via the backward-CROWN fusions below; spec16 `unsat` 1376 s, spec17 `error` (forward-zono OOM at 100× ε) — neither fits. **ABC 2026 (results.csv): spec15 1082 s; spec16 1243 s, spec17 1260 s — both also EXCEED 1200 s.** Both clear only spec15 under strict scoring. |

spec0–14 perturb only 1–100 pixels → the input box has few varying dims →
input-split + forward-zono is the right regime. spec15/16/17 perturb the whole
image; forward-zono is structurally infeasible there (see below).

## Production path (the 15/18 — landed)

The enabler is **`maxpool_to_relu`** (`onnx_optimizer.py`, gated
`settings.maxpool_to_relu`, default **True**): no backend has a real MaxPool
handler, so MaxPool is decomposed **exactly** via `max(a,b)=a+ReLU(b−a)` — a
1-hot phase-extraction Conv (C→P·C, P=kH·kW, stride=pool stride, phase-major) +
contiguous-channel Slices + a binary-max tree of Sub/ReLU/Add. Validated vs
`F.max_pool2d` (`tests/test_maxpool_to_relu.py`) and faithful to onnxruntime at
2.86e-6 on the full net.

Each decomposition op had to be taught to the paths that reject it: the gg
serializer (`verify_graph._serialize_gg_ops` += `sub_bilinear`), IBP
(`_ibp_forward_graph` += slice/gather/sub_bilinear/concat), and — the OOM fix —
**patches-native `slice`/`sub`** on `PatchesZonotope` (`slice(flat_idx)` for a
contiguous channel block stays in patch form; `sub`/`_negated` reuse `add`'s
offset-align/pad) so the maxpool decomposition never materialises the dense
`(n_flat, K)` generator matrix (was 1709 GiB). Tests: `tests/test_patches_maxpool.py`.

The forks the decomposition adds also stop `verify_graph` auto-routing conv
graphs to the milp/gpu_layers path (no MaxPool handler there);
`configs/vggnet16_2022.yaml` pins `auto_route_milp_for_conv: false`.

Soundness: a universal `sat`-witness re-validation in `main._verify` (ORT-CPU,
input-box ±tol AND output-spec ±tol) covers every verifier before emitting `sat`.

## The 3 full-image cases — why forward-zono can't, and what ABC does

**Forward-zono is structurally infeasible for full-image** (measured): the
143 000 generators (one per perturbed pixel) each grow a receptive-field patch;
by block5 the patch tensor is ~100 GB (confirmed: 128 GB CPU OOM). Box
order-reduction (`PatchesZonotope.box_reduce`) caps memory but **strips
cross-neuron correlation**, so a per-neuron box is fundamentally loose — and a
generator-magnitude probe showed the distribution is **flat** (top-100 of 143k =
0.2 % of the L1 mass), so top-m / capped-dim reduction can't preserve correlation
either (the perturbation is uniform, so all generators are near-equal). This is
exactly why ABC's `customized_vggnet16_tuning` (custom_adhoc_tuning.py) overrides
`bound_prop_method → 'crown'` (plain backward CROWN) for >100 perturbed pixels.

**ABC's recipe (measured dynamically, 2026-06-18):** plain backward CROWN, fixed
relaxations (no α), no BaB ("verified with init bound"), attack-first. **No
torch.compile, no JIT, TF32 explicitly OFF, eager fp32** (`abcrown.py:76`). Its
intermediate bounds are CROWN-backward for the deep convs (IA only for the first
conv, where it's exact). Per-layer "unstable" candidate sets are large
(19→67→559→3022→36175→268807→391197→401408 by depth — ABC over-selects too) and
it CROWN-refines them; sustained rate ~12 000 neurons/s at mid layers, dropping
to ~3 100/s at the deepest (391 197 unstable = 126 s for that one layer).

## Full-image backward-CROWN engine (`crown_retighten_*`, enabled via `--set`)

The minimal config keeps these OFF (sparse specs use input-split + forward-zono); the
full-image specs enable the patches-mode backward-CROWN retighten via the `--set`
overrides in the Repro section. History of how it got to in-budget follows.

To match ABC we built **patches-mode backward CROWN** (`src/vibecheck/patches_crown.py`)
— the backward dual of `PatchesZonotope`: the CROWN adjoint is a per-spec localized
patch propagated by `conv_transpose2d`, retightening unstable ReLU pre-activations
during the forward (`verify_zono_bnb` gated `crown_retighten_forward` /
`crown_retighten_patches`, default-off). **Validated bit-equivalent to the dense
`_crown_backward_matrix`** (conv7/10/12, rel-err ≤1.3e-7 = fp32 drift) — so it is
SOUND and, retightened fully, gives ABC-tight bounds.

### The doc's old "10–14× slower" was a mis-comparison (corrected)
ABC's per-layer "Batched CROWN" rate **collapses with depth** (measured from its
spec15 log): 11111 it/s (shallow input-40) → 5798 → 2487 → 1430 → 905 → **252 it/s
at the deepest** (input-88, 81 587 unstable = 322 s). VC's deepest conv (conv12) is
~107 n/s after the speedups below — so the gap is **~2.4× at the deepest layers**,
not 10× everywhere. The old number compared VC's deep rate to ABC's *mid* rate.

### Speedups landed (Phase A — sound, in `patches_crown.py`)
- **Zero-pad gather** (`_window_index`+`_gather_at`): out-of-range patch positions
  index a sentinel zero column → no separate mask multiply; gather one (C,H,W) map
  at a time instead of a `cat([eff_slope,up_s,up_t])` 3C tensor (the peak-memory hog
  that forced the deep chunk to OOM-collapse). + eager free + `addcmul` fuse.
  → peak mem **halved** (conv10 14.6→7.1 G), conv10 **222→250**, conv12 **50→107 n/s**.
- **Rejected**: a combined lb+ub single walk (`one_hot_both`) — it ~doubles peak mem
  (both directions live at once) and the deep layers are mem-bound, so conv12
  OOM-thrashed 107→45 n/s. Reverted to two sequential walks.
- Bigger chunks DON'T help (conv10 compute-bound; conv12 mem-capped at chunk 256).

### Why it still can't fit 1200 s (the real, measured blocker)
The forward zono is box-reduced to survive memory, which makes its bounds **explode**
through the deep layers (output box ~1e8). The final spec-backward CROWN relaxes ALL
ReLUs using `sb`, so **every un-retightened unstable neuron contributes its exploded
box bound** → garbage. Capping the retighten to fit budget therefore fails outright:
**cap=10000 → worst = −1.5e8 (verified 0/1)**, not slightly loose. You must retighten
~ALL unstable deep neurons (= ABC's pure backward CROWN). That volume —
conv10/11/12 ≈ 290k + the **block5 maxpool sub-ReLUs 75 264** (25 088 each ×3; the
block5 maxpool is FULLY unstable, and the `maxpool_to_relu` decomposition makes it
**3 sub-ReLUs** vs ABC's 1 native maxpool op) — at VC's ~107–250 n/s is **≈2900 s**,
~2.7× over the 1200 s budget. ABC does the same in 1081 s.

So: **no setting fits BOTH the budget AND the tightness needed.** Uncapped DOES
verify — **measured: spec15 → `unsat`, worst = +5.3291 (== ABC's +5.329 margin),
Phase 1+2 retighten = 2109.9 s, ~2170 s total** on the A10G. That is SOUND and
correct, but ~1.8× over the 1200 s budget (ABC: 1081 s). Any budget-fitting cap
explodes the bound (cap=10000 → −1.5e8). Production config is therefore kept
minimal — VC ships **15/18**, engine gated off, with the verification confirmed
reachable (just too slow today).

Per-layer cost (uncapped, est. from counts × measured rates): the deep block5
layers dominate — conv10/11/12 (~290k unstable) ≈ 1900 s and the block5 maxpool
sub-ReLUs (75 264) ≈ 500 s; shallow + block4 ≈ 130 s. Even stacking every plausible
engine win (maxpool-native −330 s, FC-skip −120 s, relu-fusion ~1.2× on the convs)
lands ≈ 1400 s — still over budget, because the residual blocker is the deep-conv
per-neuron rate (~1.7× slower than ABC), which Phase A's memory wins do not address.

### torch.compile the relu-backward core (the breakthrough, 2026-06-25)
The relu backward (~53 % of deep-conv time) is a chain of small elementwise kernels
(gather + clamps + products) over the (B,C,ph,pw) patch — exactly where AB-CROWN's
single fused slope-multiply beats the eager chain. `torch.compile`-ing it
(`patches_crown._relu_bwd_core`, bit-equivalent vs dense, rel-err ≤2.2e-6) gives
**conv10 270→478, conv12 88→206 n/s (1.8–2.4×)**. End-to-end uncapped spec15:
**Phase 1+2 = 1295 s, verified, worst=+5.3291** (== ABC), total ~1330 s — down from
2110 s. Memory: the compiled core's transient peak conflicts with the heavy
pre-box-reduce forward relus (143k gens), so compile is **gated to AFTER box_reduce**
(`_fwd_box_reduced` in verify_zono_bnb) — early layers eager (cheap), deep layers
compiled (forward already small). The patch-sparsity probe ruled out a native-maxpool
win (patch stays ~half-dense: the 3×3 convs re-densify between pools).

### Backward-CROWN maxpool fusions — the close (2026-06-25, `patches_crown.py`)
`torch.profiler` on the deepest layer (L23 conv11, compile ON) showed the 1295 s was
**GPU-bound, not launch-bound** (`cudaLaunchKernel` only 2 %): conv_transpose **43.6 %**
(winograd — `cudnn.benchmark` gave **1.00×**, already optimal), `aten::add` **15.8 %**
(maxpool-tree joins on big patches), the compiled relu core 15.0 %, and **~14 % pure
data movement** (zero-fill / copy / clone / neg) from the `maxpool→relu` decomposition's
backward. So two SOUND, **bit-identical** fusions of that decomposition's backward:

- **`_FAST_MAXPOOL_BWD`** — `backward_add_merge` shares the patch (every backward op
  REASSIGNS `patches`, never mutates in place, so the two merge branches need no clone);
  `backward_slice` returns a channel-BLOCK relation and `_accumulate` drops the *P*
  disjoint phase blocks of a maxpool into one tensor **in place** — no per-slice
  4×-channel zero-fill, no big channel-dim merge-add.
- **`_FAST_PHASE_CONV`** — the phase-extraction conv is 1-hot (`out_ch p*C+c` = `in_ch c`
  at window `(p//kW, p%kW)`), so its `conv_transpose2d` backward was running winograd to
  compute a pure **scatter**. Replaced with `new[:,:,py::sH,px::sW] = patches[:,p*C:(p+1)*C]`
  — exact, no winograd, and it removes cuDNN's non-deterministic dgrad atomics.
  `_detect_phase_conv` verifies the 1-hot pattern once per op (else falls back to the
  real conv_transpose — never mis-applied).

Both are **bit-identical to the naive path on CPU/float64** (deterministic;
`tests/test_patches_crown_fast.py`, atol=0 — proving the residual GPU diff is purely
cuDNN dgrad non-determinism) and match the dense `_crown_backward_matrix` (rel ≤ 6.8e-5
on GPU). Measured **1.21–1.25× on the deep layers** (conv11 299 → 375 n/s).

### Status: spec15 CLOSED — VC 16/18, matching ABC (strict 1200 s)
End-to-end (verdicts from `--results-file`):
- **spec15: `unsat`, total 1104 s (Phase 1+2 1069 s, was 1295 s), worst = +5.3291 ==
  ABC — UNDER the 1200 s budget.** PGD = 29 s at the profile default `restarts=100`; the
  close does **not** depend on any PGD trim.
- spec16: `unsat`, 1376 s (worst +0.9769). spec17: `error` — its 100×-ε perturbation
  OOMs the forward zonotope (a `copy()` wants 5 GiB on top of 17 GiB in use > 22 GiB
  A10G); the box_reduce budget that fits spec15/16 isn't aggressive enough, and tighter
  reduction explodes the bound. So spec16 VERIFIES over budget and spec17 doesn't fit
  in memory — **ABC also misses both** (1243 s / 1260 s timeouts). (spec17 graceful
  OOM→`unknown` handling is a separate robustness follow-up; it is a miss regardless.)

So the two fusions take VC from **15/18 → 16/18 in budget, matching ABC's 16/18** (both
clear spec0–15; both miss the 10×/100×-ε spec16/17). The deep convs remain at the
cuDNN-conv-transpose + fused-relu floor; closing spec16/17 would need a native maxpool
backward (the ~30 % the decomposition still spends on phase-conv + the sub/relu/add tree
that ABC does in one op) — a larger change, not attempted here.

### PGD safety (single-config requirement)
The competition uses ONE config per benchmark — no per-instance tuning. A reduced PGD is
only valid if it still catches every counterexample. vggnet16's only SAT case is **spec0**
(nominal misclassification); measured: at `pgd_restarts=5` spec0 still returns `sat`
(ORT-validated). So restarts=5 is a safe single-config choice (≈−26 s on every instance),
but spec15 already fits at the default restarts=100, so the close does not rely on it.

### Why spec16/17 don't close (2026-06-25 deep-dive — all measured, scratch/vgg_*.py)
spec16 verifies (unsat, +0.9769) but at 1376 s; spec17 forward-zono OOMs at 100× ε. Every
cheap lever was tested and is DEAD:
- **Tighter forward** (less box_reduce) → OOM: the forward already peaks **23.0 GB** at the
  A10G ceiling; any larger budget OOMs.
- **Skip the over-selected block4 retighten** → OOM: skipping conv9's retighten leaves it at
  the *exploded* box-reduce bounds, so its `apply_relu` floods generators and the forward
  zono blows up. The retighten is load-bearing (it keeps the forward memory-feasible), not
  wasted work.
- **Drop the zono for bigger retighten batches** → no speedup: with the forward zono freed,
  the deep retighten rate is FLAT (378→380 n/s, 256→512 chunk) — it is **compute-bound** at
  the cuDNN conv_transpose, and the 160×160-RF patches are themselves the memory hog.

The real (smaller) gap, properly measured: **VC's per-neuron deep backward is comparable to
or faster than ABC** (conv9: VC 1313 vs ABC 907 n/s, same ~92×92 RF patch) — there is NO
representation/"stride-tracking" win (both carry RF-sized patches; an earlier per-layer
comparison that suggested a 4× kernel gap was apples-to-oranges across different layers'
receptive fields). The actual difference is **selection**: box_reduce marks **81 % of conv9
unstable vs ABC's 20 %**, so VC retightens ~4× more neurons. ABC's tighter intermediate
bounds come from pure backward-CROWN (no exploding box_reduce) — but VC can't get that
selection cheaply (box_reduce explodes, a tighter forward OOMs, and the backward-CROWN that
would give a tight selection IS the expensive retighten). **And even ABC's own method misses
the budget: ABC spec16 = 1243 s > 1200 s.** So spec16/17 are a miss for *everyone* under
strict scoring; closing them is an open problem, not a config or kernel tweak.

## Repro

Sparse (spec0–14) — minimal config:
```bash
.venv/bin/python -m vibecheck.main <onnx> <vnnlib> --config configs/vggnet16_2022.yaml --results-file out.txt
```

Full-image (spec15–17) — enable the backward-CROWN retighten + forward-only cascade
(the cascade defaults would record every relu's pre-rows and OOM at the deep layers):
```bash
.venv/bin/python -m vibecheck.main <onnx> <vnnlib> --config configs/vggnet16_2022.yaml \
  --device gpu --timeout 1500 --results-file out.txt \
  --set crown_retighten_forward=true --set crown_retighten_patches=true \
  --set box_reduce_patch_budget=6.0e+9 --set box_reduce_on_oom=true \
  --set phase1_method=plain --set tighten_formulation=skip --set max_tighten_layer=0 \
  --set crown_retighten_chunk=64
```
(`crown_retighten_compile` defaults true; the `_FAST_MAXPOOL_BWD`/`_FAST_PHASE_CONV`
fusions are module-default ON. `scratch/run_spec15_fast.sh` wraps this.)

Sweep + diagnostics: `scratch/vgg_*.py` (chunk sweep, conv microbench, per-layer
breakdown, ABC op-profile) — all gitignored scratch. ABC reference traces:
`~/persistent_runs/abc_trace/`, `abc_spec15/` on the g5 box.
