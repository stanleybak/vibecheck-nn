2. Current pipeline structure
------------------------------------------------------------------

Phase 1 (interleaved zono forward + tightening):
  for each op in topological order:
    if op is not a relu: propagate zonotope normally
    if op is a hidden relu (li > 0):
      1. read zono's internal bounds lo_np, hi_np
      2. run _per_neuron_adaptive_bounds over unstable_initial
         (backward CROWN with min-area slopes, tightened upstream
         bounds feed in via bounds_by_relu[0..li-1])
      3. intersect: new_lo = max(zono, adapt), new_hi = min(zono, adapt)
      4. if has-merge-upstream: _tighten_layer_graph_sparse
         (per-target-neuron sparse model, MILP probe → LP fallback,
         sticky 'probe' → 'lp' → 'skip' mode)
         else: _tighten_sequential_with_probe (existing
         _tighten_layer_parallel with sparse conv MILP)
      5. intersect: new_lo/new_hi with lp_lo/lp_hi
      6. z.apply_relu(tight_lo=new_lo, tight_hi=new_hi)
         — zonotope's relu uses the intersected bounds for slope,
         so the downstream zono state is tighter too.

Phase 2: spec CROWN backward (_spec_backward_graph, uses tightened
         bounds_by_relu via triangle slopes)
Phase 3: PGD attack (runs on a dedicated float32 gg_pgd — keeps PGD
         cheap even when the main pipeline is float64)
Phase 4: REMOVED (merged into phase 1)
Phase 5: REMOVED (was a redundant CROWN recheck — phase 2 already
         uses the interleaved-tightened bounds)
Phase 6: REMOVED (redundant PGD — phase 3 already attacks the
         tightened network)
Phase 7a: parallel LP feasibility pre-check across all open queries
          (via _solve_spec_worker_graph with mode='feasibility',
          Pool sized to n_queries, each worker gets n_cores/n_queries
          threads). LP-infeasible → query verified.
Phase 7b: cheap CROWN-ew × width fallback scoring (no LP, since full-
          LP scoring was hitting timeouts and returning empty dicts).
Phase 8: per-query racing escalation (_racing_escalation_graph_correct),
         doubling bin schedule [0, 2, 4, 8, …], three-way dispatch
         (SAT / UNSAT / UNKNOWN).
Phase 9: final PGD (guards the path out on timeout).

3. Key soundness fixes
------------------------------------------------------------------

(a) Bug #1 in verify_milp._solve_spec_graph_worker — dead-branch bias
    was being dropped (out[j] = None when expr.size() == 0). The new
    version emits a fixed-bound `bias[j]` variable and downstream
    adds / the spec expression pick it up. Regression test
    test_verify_milp_worker_drops_dead_bias_regression.

(b) _per_neuron_adaptive_bounds upper-bound was wrong — the first
    version tracked only one EW and negated at the input, which is
    mathematically incorrect for upper bounds. Now tracks both
    EW_lb and EW_ub with different slope choices:
      lower: ew_lb = ep*lo_s + en*up_s, acc_lb += (en*up_t).sum
      upper: ew_ub = ep*up_s + en*lo_s, acc_ub += (ep*up_t).sum
    (ported from verify_zono_bnb._evaluate_region lines 599-675).
    A 10,000-sample Monte Carlo test on the sequential FC network
    confirms soundness.

(c) Default bits=32 → bits=64. float32 GPU zonotope was producing
    *unsound* under-approximations on CIFAR100 (the LP for li=2 was
    strictly infeasible with the float32 bounds, but feasible with
    float64). PGD is still float32 via a dedicated gg_pgd.

5. What's known to be slow
------------------------------------------------------------------

Phase 1 spends 115 s, most of it on layers 2 / 3 / 5 where the graph
sparse cone explodes:

  - li=2: cone covers both branches of one residual block → ~10k var
    sparse model per target neuron. Probe succeeds in 1.7-5 s but
    full solve is 20 s because worst-case neurons hit the 5 s
    per-direction timeout × 35-146 targets / 16 cores.
  - li=3: cone covers two residual blocks → larger model. LP fixes
    zero neurons but burns 44.81 s. The single-neuron probe at li=3
    was FAST (1.7 s) — it's not a predictor of worst-case neurons.
  - li=5: cone spans four residual blocks → build alone is 8.38 s
    (exceeds sample_timeout=5 s), layer correctly skipped but the
    build time was wasted.

Phase 7a: 3 parallel LP feasibility checks, each using a big full-spec
model (~110 k vars / 94 k constraints). Each times out at ~30 s
returning UNKNOWN — not INFEASIBLE, so not enough to verify.

Phase 8: qi=25 at bins=0 racing got `opt lb=-0.6066 (22 s) → escalate`,
then bins=2 feasibility started and ran out of time. Effectively,
only 1 of 3 queries gets a MILP attempt, never reaches a useful bin
level.

      solve took real time. Minimal heuristic, matches user's earlier
      guidance ("if probe times out, skip later layers").

      3 queries; each query's racing is internal-parallel (feas + opt).
      With 3 queries × Pool(2), MILP gets 3× more wall time per query.
  (d) Add the zonotope-direct spec bound alongside _spec_backward_graph
      and take max(zono_direct, crown) per query. This was DISCUSSED
      but NOT added — the user confirmed they're not mathematically
      equal (parallelogram vs triangle), so the "free tighter bound"
      idea only helps on networks where correlation-preservation
      beats triangle slopes.
